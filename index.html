Step-by-Step Technical Guide
1. User → Merchant 3DS ClientUser enters card details on merchant 3ds client site.
2. Merchant 3DS Client → Merchant 3DS ServerMerchant 3DS Client triggers Merchant 3DS Authentication.
3. Merchant 3DS Server → DSAuthentication Request sent to DS.
4. DS → ACSDirectory server forwards AReq to ACS.
5. ACS → CMSACS sends AuthReq to CMS.
6. CMS → ACSCMS returns Authorisation Response.
7. ACS → DSACS returns ARes to DS with acsURL.
8. DS → Merchant 3DS ServerDS returns final Authentication Response.
9. Merchant 3DS Client → ACSMerchant 3DS client sends CReq (Challenge Request) to ACS.
10. ACS → Merchant 3DS ClientThe ACS renders a OTP page, and within this page, the Initial CRes is conveyed to the merchant’s 3DS client.
11. ACS → CMS or OTP/NotificationACS triggers CMS or OTP/Notification according to the client's decision.
12. User → OTP PageUser enters OTP in the OTP Page.
13. OTP Page → ACSUser sends verify OTP request from the OTP Page to ACS.
14. ACS → DSThe ACS sends an RReq (Result Request) to the Directory Server (DS).
15. DS → Merchant 3DS ServerThe Directory Server (DS) forwards the RReq to the Merchant 3DS Server.
16. Merchant 3DS Server → DSThe Merchant 3DS Server processes the request and returns an RRes (Result Response) to the Directory Server (DS).
17. DS → ACSThe Directory Server (DS) then forwards the RRes back to the ACS .
18. ACS → Merchant 3DS ClientFinal Challenge Response (CRes) sent to client. If payment is successful Merchant 3DS Client will show a SUCCESS page or else FAILED page.
